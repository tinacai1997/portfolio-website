Lingually

Your immersive language learning application

Overview

Lingually is an educational app that encourages users to learn new languages through fun challenges, custom reminders, and adaptable paces.
Lingually was made to work for users with a variety of schedules and studying habits, making it suitable for users with tight schedules, or those that need quiet time.
Role: UX/UI Designer (solo)
Tools: Adobe XD, Marvel, Google Sheets
Timeframe: May 2021 - June 2021

Understanding the problem ðŸ§ 

Competitive analysis

Before Lingually became a language learning app, it began as a vocabulary app. Though the content was different, the core problem was comparable: a mobile app that motivated learners to continuously practice.
To get an idea of existing vocab apps, I evaluated Atlas, Magoosh, and Vocabulary. From these analyses, I produced summaries and lists of likes and dislikes to guide Lingually's design. My key takeways from these analyses were:
  • Each app did a good job of remaining focused on their target audiences based on UI and content.
  • The apps with interactive challenge systems were better at retaining my attention.
  • I had a harder time grasping new words in the apps that didn't provide revisiting or audio features.

User interviews

I interviewed vocabulary learners to gain insight on how they approached learning and the educational tools they prefered. I gathered the interview data by separating my observations into three categories:
DOING
THINKING
FEELING
This is when Lingually diverged from a vocab app to a language learning app. All participants used language apps over strictly vocabulary apps and grouped learning languages and vocabulary as one concept. Participants generally focused on their experience with learning new vocab in non-native languages when answering questions.

Empathizing with users ðŸ’ž

User personas

From the list of Doing, Thinking, and Feeling insights, I created the persona Tony Martinez, who gave a face for users with needs that Lingually aimed toward solving.
When designing Lingually, it was helpful to think back on how my solutions helped solved Tony's problems, rather than a list of problems.

User stories

My next step was to draft user stories for Tony. These stories helped me further emphasize with him by describing his needs in his perspective.
HANDS ON
HABITUAL
PRONUNCIATION
  • As a hands-on learner, I want interactive learning activities so that I can truly practice what I've learned rather than just reading.
  • As a habitual learner, I want tool that helps me keep up my learning progress so that I can make studying a daily habit.
  • As a language learner, I want to be able to hear words spoken out by native language speakers so that I can feel confident about my own grammar and pronunciation.

Problem & hypothesis statements

Armed with a strong understanding of who I'm desiging for, I created a problem statement to focus my solution toward and a hypothesis statement that predicts Lingually's outcome assuming it is designed toward Tony's needs.
Problem: Tony needs a fun and interactive way to improve his Japanese that can adapt to his schedule because he wants to better absorb Japanese media and be proficient in basic Japanese when he travels to Japan next year.
We will know this to be true when we see him build a regular habit of studying Japanese every day, and Tony is able to hold a basic conversation in Japanese.
We believe that by creating an interactive learning system that offers features like quiz games, pronunciation practice, and realistic example conversations, we will achieve an immersive study tool that motivates Tony to practice Japanese.

User flows

I created user flows to represent Lingually functions that could meet Tony's needs and fulfill the hypothesis statement. Each user flow presents a solution to Tony's core needs in the form of three key-functions: create an account, set a reminder, and complete a challenge.

Low Fidelity Wireframes

Using Adobe XD, I "sketched" lo-fi wireframes for Lingually that act as the UI for the user flows. In hindsight, I should have sketched these lo-fi wireframes using pencil and paper.
  • With XD, I was too focused on pixel-perfection. The purpose of lo-fi wireframes are to represent functionality; striving for pixel-perfection defeated that purpose.
  • I couldn't see all my ideas. I could freely sketch with pencil and paper, but with XD, my old ideas were hidden by newer ones.

Testing the prototype ðŸ§ª

Usability testing

I used the web app, Marvel, to link my lofi wireframes together and create an interactive prototype. I then conducted usability tests using the prototype with participants like Tony. The purpose of the tests were to assess whether users could complete the following tasks: create an account, complete a lesson, create a flashcard, and set a reminder.
I put together a usability test report with the data from the tests. This report summarized participant errors and their severity for improving Lingually.

Error addressal

I addressed the errors in the usability test report by iterating on my lofi wireframes in order of severity (in this case, they were all level 3).
Issue #1: Onboarding bubbles weren't obviously onboarding bubbles.
Severity: High
Evidence: Two of three participants did not notice onboarding bubbles.
Solution: Darken background and onboarding bubble button when onboarding content is focused.
Additional Updates: Some participants mentioned that they usually skip onboarding. Added a skip button for experienced users or users that want to learn on their own.
Issue #2: Users had trouble setting up a reminder.
Severity: High
Evidence: All users had trouble finding where to set up reminders and only found where to do so through deduction.
Solution: Mention reminders during onboarding and change the "Profile" tab to "Profile & Settings"
Issue #3: Users were confused with how to create a new flashcard deck.
Severity: High
Evidence: Two of three participants looked around for a while before finding the add a deck button.
Solution: Change add a deck button from a plus sign to a bar button with text and darken its color.
Additional Updates: Some participants found the sample decks distracting. Added a prompt to encourage new users to create decks in place of sample decks. Also added an edit button for existing decks.

The takeaways ðŸ¤”

Challenges

  1. Interview participants: Not a challenge per se, but interview participants were primarily language app users rather than vocabulary. It made sense to adjust the project scope based on this outcome, but whether I'm able to do so would depend on client goals if there were any. I anticipate I'd have a larger pool of participants to test with when working at a company, but if not, it would be a matter of communicating with the client about changes like this.

  2. Lofi wireframes: As mentioned earlier, I spent too much time making pixel-perfect lofi wireframes using Adobe XD. Next time, I will put that energy into ideating with pencil and paper.

  3. Usability testing: It was difficult to keep up note-taking while also facilitating the usability tests. During future tests, I will use screen recording software and create recording consent forms to track observations.

  4. Testing lo-fi wireframes: I noticed during usability tests that the lack of content and color and abundance of "squiggles" in my lo-fi wireframes distracted users from completing tasks. In the future, I would either decrease "lo-fi disctractions" or use higher fidelity wireframes for usability testing.

Next steps

  1. Mid-fi wireframes: Lingually was a great first project to start on as it hit all the key UX research methodologies I would use while working on other case studies. My next steps for Lingually are to create midfi wireframes to iterate on the app's UI based on all the feedback I've gathered up to this point.
