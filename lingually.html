Lingually

Your immersive language learning application

problem

How might we design a mobile app that empowers people to learn new languages?

Language learners want to learn new languages for a variety of reasons, but it's hard to fit time in their busy schedules and stay motivated in practicing.
The goal of this language learning app is to provide an immersive educational tool that reminds language learners to practice at their own pace.

The research üß†

Competitive analysis

What makes a good educational tool?

Before Lingually became a language learning app, it began as a vocabulary app. Though the content was different, the core problem was comparable: a mobile app that motivated learners to continuously practice.
To get an idea of what successful vocab apps looked like, I evaluated three existing ones: Atlas, Magoosh, and Vocabulary. From these analyses, I produced a summary and list of likes and dislikes.
See Competitive Analyses

User interviews

How do users learn new languages and vocabulary?

I interviewed vocabulary learners to gain insight on how they approached learning and the educational tools they prefered.
I gathered the interview data by separating my observations into three categories: Doing, Thinking, and Feeling. It was at this point when Lingually diverged from a vocab app to a language learning app. During interviews, target-users grouped learning languages and vocabulary as one concept, and all shared their experiences with learning new vocabulary in their non-native languages. It made sense to make this change at this early stage of the project to design toward my users' needs and goals.
See User Interview Data

User personas

What are the needs, goals, and behaviors of Lingually's typical user?

From the list of Doing, Thinking, and Feeling insights, I created the persona Tony Martinez, who acted as a user with needs that Lingually now aimed toward solving.
This persona was useful because it gave a face and identity to my typical target-user. After all, it's much easier to design with a person in mind than a list of observations.

user stories

Using the interview data again, I drafted user stories for Tony. These stories helped me further emphasize with him by describing his needs in his perspective.
  1. As a hands-on learner, I want interactive learning activities so that I can truly practice what I've learned rather than just reading.
  2. As a habitual learner, I want tool that helps me keep up my learning progress so that I can make studying a daily habit.
  3. As a language learner, I want to be able to hear words spoken out by native language speakers so that I can feel confident about my own pronunciation.

    job stories

    I then drafted job stories for Tony, which were similar to user stories but honed in on specific scenarios. This task was useful because it sparked ideas for the app's functionality. These user and jobs stories led to an overarching problem and hypothesis statement.

    1. When I am watching something in Japanese, I want to be able to pick up words and phrases from hearing so I can rely on English subtitles less often for understanding what I'm watching.
    2. When I am speaking to a native Japanese speaker, I want to confidently hold a basic conversation with them so I can create a genuine interaction.
    3. When I am dedicating time to study, I want to experience various learning methods so I can continuously work on tasks and not feel they are redundant.

    problem statement

    Tony needs a fun and interactive way to improve his Japanese that can adapt to his schedule because he wants to better absorb Japanese media and be proficient in basic Japanese when he travels to Japan next year.
    We will know this to be true when we see Tony build up a regular habit of studying Japanese every day, and Tony is able to hold up a basic conversation in Japanese.

    hypothesis statement

    We believe that by creating an interactive learning system that offers features like quiz games, pronunciation practice, and realistic example conversations, we will achieve an immersive study tool that motivates Tony to practice Japanese.

    The structure üß±

    User flows

    How might users utilize a language learning app to meet their goals?

    The aim of Lingually's design was now to focus on solving Tony's specific problem statement. To do so, I created user flows to represent Lingually functions that could meet Tony's user and job story needs. If Lingually is designed with these in mind, it should fulfill the hypothesis statement.
    Each user flow presents a solution to Tony's core needs in the form of a key-function.
    1. Need: to save his statistics and information.
      Solution: Login/Create an Account
    2. Need: to ensure he is practicing Japanese on a regular schedule.
      Solution: Set Reminders
    3. Need: to progress learning Japanese in a fun and interactive way.
      Solution: Complete a Challenge

    Low Fidelity Wireframes

    Let's focus on the basic functionalities of Lingually.

    Using Adobe XD, I "sketched" lo-fi wireframes for Lingually that act as the UI for the user flows. In hindsight, I should have sketched these lo-fi wireframes using pencil and paper for several reasons.
    1. With XD, I was too focused on "pixel-perfection". However, the purpose of lo-fi wireframes are to represent functionality, it wasn't worth the time trying to make them look perfect.
    2. I couldn't see all my ideas and mistakes. With pencil and paper, I could freely sketch. With XD, my ideas were hidden by the newest ones.

    Prototype testing üß©

    Usability testing

    How can Lingually be improved to successfully meet Tony's needs?

    I used the web app, Marvel, to link my lofi wireframes together and create an interactive prototype. I then conducted usability tests using the prototype with participants like Tony.
    The purpose of the tests were to assess whether users could complete specific scenarios. The scope of the test included: Creating an Account, Completing a Lesson, Creating a Flashcard, and Setting Up a Reminder.
    See Lofi Prototype

    usability test report

    Given the list of observations from the usability tests, I put together a usability test report. This report summarized participant errors, and therefore areas in which Lingually could improve.

    Design refinement üî®

    error addressal

    I addressed the errors I uncovered in the usability test report by iterating on the lofi wireframes in order of severity.
    Issue #1: Onboarding bubbles weren't obviously onboarding bubbles.
    Severity: High
    Evidence: Two of three participants did not notice onboarding bubbles.
    Solution: Darken background and onboarding bubble button when onboarding content is focused.
    Additional Updates: Some participants mentioned that they usually skip onboarding. Added a skip function for those that want to learn on their own or experienced users.
    Issue #2: Users had trouble setting up a reminder.
    Severity: High
    Evidence: All users had trouble finding where to set up reminders and only found where to do so through deduction.
    Solution: Mention reminders during onboarding and change the "Profile" tab to "Profile & Settings"
    Issue #3: Users were confused with how to create a new flashcard deck.
    Severity: High
    Evidence: Two of three participants looked around for a while before finding the add a deck button.
    Solution: Change add a deck button from a plus sign to a bar button with text and darken its color.
    Additional Updates: Some participants found the sample decks distracting. Added a prompt to encourage new users to create decks in place of sample decks. Also added an edit button for existing decks.
    Issue #4: A user had trouble understanding what to do during the select a language screen.
    Outcome: This was a result of the lofi wireframes using minimal text and has been ignored as an actual functionality change.

    The takeaways ü§î

    Challenges

    • Interview participants

      Not a challenge per se, but interview participants were primarily language app users rather than vocabulary. It made sense to adjust the project scope based on this outcome, but whether I'm able to do so would depend on client goals if there were any. I anticipate I'd have a larger pool of participants to test with when working at a company, but if not, it would be a matter of communicating with the client about changes like this.

    • Lofi wireframes

      As mentioned earlier, I spent too much time making pixel-perfect lofi wireframes using Adobe XD. Next time, I will put that energy into ideating with pencil and paper.

    • Usability testing

      It was difficult to keep up note-taking while also facilitating the usability tests. During future tests, I will use screen recording software and create recording consent forms to track observations.

    next steps

    • Mid Fidelity Wireframes

      Lingually was a great first project to start on as it hit all the key UX research methodologies I would use while working on other case studies. My next steps for Lingually are to create midfi wireframes to iterate on the app's UI based on all the feedback I've gathered up to this point.
